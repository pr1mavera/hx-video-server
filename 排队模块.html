排队模块

华夏视频服务 SDK 文档

排队模块的通信基础建立在IM模块之上的，整体的设计思路是通过外部提供的排队参数的格式化参数（包括排队控制器参数，接口适配器，生命周期钩子函数等），自动构建起整个排队自动化流程。

# 排队模块初始化

排队模块初始化需要传入三个参数

  • IM - 外部初始化好的IM实例
  • controller:Object - 排队控制器参数
  • listener:Object - 生命周期钩子函数

初始化完成将会返回两个API

  • doQueue - 执行排队
  • cancel - 取消排队

整个初始化过程即为生成这两个API，熟悉 rx 的开发者应该清楚，任何 Observable 数据的流动都是需要在订阅之后，因此这里的构建过程也就是数据流订阅与取消订阅的过程
即 doQueue 是将一系列数据流与注册的生命周期钩子包装、合并、组合、订阅，并且各个数据流之间通过事件总线互相通信 cancel 则是将订阅的数据流取消订阅

// 订阅器集合，须在取消排队时注销所有订阅
let unsubs = [];

const doQueue = () => {

    // 合并数据管道，统一订阅
    unsubs.push(merge(socket$, queue$, heartBeat$, timeout$, cancelQueue$).subscribe(
        // 这里直接将 事件总线实例 直接作为observer进行注册观测
        action$
    ));

    // 订阅排队成功时的hook
    unsubs.push(action$.pipe(
        // 过滤出排队成功的数据流
    ).subscribe(
        // 执行 onQueueFinish 钩子
    ));

    // 订阅排队失败时的hook
    unsubs.push(action$.pipe(
        // 过滤出排队失败的数据流
    ).subscribe(
        // 执行 onQueueFailed 钩子
    ));

    // 订阅排队接口返回数据的hook
    onAfterRequestQueue && unsubs.push(action$.pipe(
        // 过滤出调用排队接口返回数据的指令
    ).subscribe(
        // 执行 onAfterRequestQueue 钩子
    ));

    // 订阅系统消息推送的hook
    onSendSystemMsgSucc && unsubs.push(send$.subscribe(
        // 执行 onSendSystemMsgSucc 钩子
    ));

    // 订阅人数减少时的hook
    onQueueNumDecrease && unsubs.push(action$.pipe(
        // 过滤出人数减少的指令
    ).subscribe(
        // 执行 onQueueNumDecrease 回调
    ));

    // 订阅心跳超时限制的hook
    onHeartBeatOverLimit && unsubs.push(action$.pipe(
        // 过滤出心跳超时限制的指令（由于后续的排队机制优化，现已弃用）
    ).subscribe(
        // 执行 onHeartBeatOverLimit 钩子
    ));

    // 订阅排队成功客户端握手hook
    onQueueHandShake && unsubs.push(action$.pipe(
        // 过滤出排队成功客户端握手的指令
        filter(({ QUEUE_HANDSHAKE }) => QUEUE_HANDSHAKE),
    ).subscribe(
        // 执行 onQueueHandShake 回调
    ));

    // 统一绑定总线通知
    unsubs.push(action$.subscribe(() => {}, e => onQueueFailed(e)));
};

/**
 * 取消排队 | API
 */
const cancel = () => {
    action$.next({ 'CANCEL_QUEUE': { isCancelByError: false } });
    // 因为取消排队的指令消息推送是在下一个宏任务中产生的，因此这里取消所有流的订阅的操作需要延迟处理
    setTimeout(() => {
        // 订阅器清空
        unsubs.forEach(unsubscribable => {
            unsubscribable.unsubscribe();
        });
        unsubs = null;
    }, 20);
};

# 排队控制器参数

排队控制器参数用于定义排队接口相关、心跳接口相关、超时相关的一系列结构化配置信息，具体的参数格式请看代码注释
排队模块初始化过程中也是通过解构的方式将其中的配置项解析出来

const {
    /**
     * 调用排队接口的封装好的Promise，format
     * 
     * {
     *     entry: () => API(),
     *     format: input => output
     * }
     */
    doQueueAPI,

    /**
     * 调用取消排队接口的封装好的Promise，format
     * 
     * {
     *     entry: () => API(),
     *     sendData: data
     *     format: input => output
     * }
     */
    cancelQueueAPI,

    /**
     * 调用心跳接口的封装好的Promise，format
     * 
     * {
     *     entry: () => API(),
     *     format: input => output,
     *     rate: 7000,
     *     failCount: 2
     *     limitCount: 20
     * }
     */
    heartBeatAPI,

    /**
     * 超时字段
     * 
     * {
     *     limit: 1000 * 3
     * }
     */
    timeout,

    /**
     * 排队服务需要接收的IM推送的系统信息集合
     * key:       消息体的类型，code
     * value:     [ 该类型的消息代表的排队状态的事件、作为事件总线中事件类型, 格式化输出 ]
     * 
     * '20': [ 'QUEUE_DECREASE', format ],          // 人数减少通知
     * '21': [ 'QUEUE_SUCC', format ],              // 客户排队成功通知
     * '23': [ 'CS_RESPONSE_SUCCESS', format ],     // 坐席响应成功通知
     * '26': [ 'CS_RESPONSE_FAILED', format ]       // 坐席响应失败通知
     */
    _IM_rsv,

    /**
     * 排队服务需要发送的系统消息集合
     * 
     * 'QUEUE_SUCC': 22,        // 发送 -> 排队成功一次握手，入参为调用排队接口的出参
     * 'QUEUE_TIMEOUT': 27,     // 发送 -> 排队超时，入参为调用排队接口的出参，或排队成功IM推送的参数
     * 'QUEUE_CANCEL': 28,      // 发送 -> 取消排队，参数为外部传入
     */
    _IM_send

} = controller;

# 生命周期钩子函数

同样，生命周期钩子函数也通过初始化入口注册进来，会在合适的时机与合适的数据流结合

// 注册事件监听
const {
    // 调排队接口，传递调用结果
    onAfterRequestQueue,
    // 队列人数减少通知
    onQueueNumDecrease,
    // 当排队完成（包括 调排队接口直接完成 和 接收到推送的排队完成系统消息），并发送完成通知坐席的消息
    onSendSystemMsgSucc,
    // 排队成功通知 !!! require !!!
    onQueueFinish,
    // 排队失败总通知，返回失败信息（包括排队API报错、心跳报错），将导致排队终止 !!! require !!!
    onQueueFailed,
    // 心跳超出次数限制，响应通知
    onHeartBeatOverLimit,
    // 订阅排队成功客户端握手hook
    onQueueHandShake
} = listener;

# 运行机制

上面说的整个初始化过程即为生成这两个API，合并订阅数据流和钩子函数，那么具体的数据流是哪些呢？

运行机制重点讨论的就是这些数据流

  • action$ - 事件总线数据流
  • socket$ - IM消息接收数据流
  • queue$ - 排队数据流
  • heartBeat$ - 心跳数据流
  • timeout$ - 超时数据流（排队流程优化后弃用）
  • cancelQueue$ - 取消排队数据流
  • send$ - IM消息发送数据流

整体流程：（注：排队流程优化后）

  1. 订阅「排队数据流」导致执行排队接口，根据返回的结果，向「事件总线」推送排队中、排队完成等指令
  2. 「心跳数据流」监控到排队中的指令，开启心跳
  3. 「IM消息接收数据流」监控到坐席发来的排队成功消息，关闭心跳，同时向「事件总线」推送发送排队成功客户端握手的消息推送指令
  4. 「IM消息发送数据流」接收到排队成功客户端握手的指令后向IM推送消息
  5. 外部响应注册的排队成功客户端握手的回调，整个排队流程成功
